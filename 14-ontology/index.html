Section 01

What is an Ontology?

A formal model of the world — concepts, properties, and relationships. Not just a list of terms, but a machine-readable specification of how things relate and what rules govern them.

Simpler

📅 Taxonomy

  • Hierarchy only
  • Parent / child classes
  • No relationships
  • No rules
  • e.g. biological classification
is-a hierarchy
Richer

🌐 Ontology

  • Hierarchy + relationships
  • Object & data properties
  • Axioms & constraints
  • Machine-readable logic
  • e.g. OWL schema
schema + rules
Full stack

📈 Knowledge Graph

  • Ontology + data instances
  • Real-world entities
  • Queryable triples
  • Scalable to millions of nodes
  • e.g. Wikidata, Google KG
ontology + data
Analogy: A taxonomy is an org chart (hierarchy only). An ontology is an org chart plus job descriptions, reporting rules, policies, and constraints — everything you need to reason about how the organisation actually works.
💡 Key insight: ontologies answer not just "what exists" but "how things relate and what rules govern them" — enabling automated reasoning.
Section 02

Core Components

Three building blocks — click each to expand.

📁
Building block 1
Classes (Concepts) — the types of things that exist
⌄

Classes define categories of real-world things. They can inherit from parent classes (subClassOf) and can be disjoint, equivalent, or overlapping.

Examples: Server, Application, Incident, Person, Service, Database, Datacenter

:Server rdf:type owl:Class . :WebServer rdfs:subClassOf :Server . :DatabaseServer rdfs:subClassOf :Server . # WebServer and DatabaseServer are disjoint :WebServer owl:disjointWith :DatabaseServer .
🔗
Building block 2
Properties (Relationships & Attributes) — how things connect
⌄

Object properties link instances to instances. Data properties link instances to literal values (strings, numbers).

Examples: hosts, dependsOn, assignedTo, hasStatus, hasIPAddress

:hosts rdf:type owl:ObjectProperty ; rdfs:domain :Server ; rdfs:range :Application . :hasStatus rdf:type owl:DatatypeProperty ; rdfs:domain :Incident ; rdfs:range xsd:string .
📖
Building block 3
Axioms (Rules & Constraints) — logical rules that govern the domain
⌄

Axioms express logical constraints that must hold. They enable automated reasoning — inferring new facts and detecting contradictions.

Examples: "Every Incident must affect at least one Service" · "A P1 Incident requires an on-call engineer" · "A Server cannot be its own host"

# Every Incident affects at least 1 Service (min cardinality) :Incident rdfs:subClassOf [ rdf:type owl:Restriction ; owl:onProperty :affects ; owl:minCardinality 1 ] . # P1 Incident subclass requires on-call engineer :P1Incident rdfs:subClassOf :Incident ; rdfs:subClassOf [ owl:onProperty :hasAssignee ; owl:someValuesFrom :OnCallEngineer ] .
Section 03

Representation Formats

How ontologies are serialised — four major formats, each with distinct trade-offs.

RDF/Turtle — Subject → Predicate → Object Triples

RDF (Resource Description Framework) is the foundational data model. Every statement is a triple. Turtle is a compact, human-readable serialisation. Use when: storing graph data, exchanging between systems, feeding triplestores.

@prefix : <http://example.org/itsm#> . @prefix rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#> . @prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> . # Instance triples :AppServer1 rdf:type :Server ; :hosts :CRMApp ; :hasIPAddress "10.0.1.42" . :CRMApp rdf:type :Application ; :dependsOn :CRMDatabase ; :isMissionCritical true .
W3C Standard Human readable

OWL — Web Ontology Language

OWL extends RDF with class expressions, property restrictions, cardinality constraints, and logical axioms. It enables automated reasoning via OWL reasoners (HermiT, Pellet). Use when: you need inference, consistency checking, or complex constraints.

# OWL class with restriction (Turtle syntax) :MissionCriticalServer rdf:type owl:Class ; owl:equivalentClass [ rdf:type owl:Restriction ; owl:onProperty :hosts ; owl:someValuesFrom [ rdf:type owl:Class ; owl:intersectionOf ( :Application [ owl:onProperty :isMissionCritical ; owl:hasValue true ] ) ] ] .
Reasoning enabled DL expressivity

JSON-LD — JSON for Linked Data

JSON-LD embeds a @context into standard JSON to give it semantic meaning. Every property maps to a URI. Use when: REST APIs, web apps, schema.org markup, LLM context injection.

{ "@context": { "@vocab": "http://example.org/itsm#", "rdfs": "http://www.w3.org/2000/01/rdf-schema#" }, "@id": "AppServer1", "@type": "Server", "hosts": { "@id": "CRMApp" }, "hasIPAddress": "10.0.1.42", "rdfs:label": "Application Server 1" }
API friendly schema.org compatible

SPARQL — SQL for RDF Graphs

SPARQL is the W3C query language for RDF triplestores. It supports SELECT, CONSTRUCT (derive new triples), ASK (boolean), and DESCRIBE. Use when: querying knowledge graphs, building graph-based RAG, generating reports.

# Find all applications on servers in Datacenter-West PREFIX : <http://example.org/itsm#> SELECT ?server ?app ?status WHERE { ?dc rdf:type :Datacenter ; :name "Datacenter-West" . ?server :locatedIn ?dc ; :hosts ?app . ?app :hasStatus ?status . FILTER(?status != "healthy") } ORDER BY ?server
Pattern matching W3C Standard

Section 06

Reasoning Over Ontologies

What makes ontologies powerful — automated inference from existing facts and rules.

🤖

Classification

Reasoners automatically assign instances to the most specific class they satisfy, even without explicit type assertions. Add an instance matching MissionCriticalServer's restrictions — it gets classified automatically.

automatic typing
⚠️

Consistency Checking

Detect logical contradictions — e.g. an instance typed as both WebServer and DatabaseServer when they're declared disjoint. Catches modelling errors before they propagate to production.

error detection
💡

Entailment

Derive new facts not explicitly stated. If "AppServer1 hosts CRMApp" and "CRMApp isMissionCritical true" → the ontology infers "AppServer1 is MissionCriticalServer" without explicit assertion.

new fact derivation
# SPARQL CONSTRUCT — derive new triples via inference PREFIX : <http://example.org/itsm#> CONSTRUCT { ?server rdf:type :CriticalInfrastructure . ?server :criticalityReason "Hosts mission-critical application" . } WHERE { ?server :hosts ?app . ?app :isMissionCritical true . # Inferred triples are written back to the graph }
Section 07

Common Ontology Patterns

Reusable design patterns — proven solutions to recurring modelling challenges.

Part-Whole Pattern

Use when things are composed of other things. Distinguished from sub-classification (is-a). When to use: hardware components, service dependencies, organisational units.

:hasComponent rdf:type owl:ObjectProperty ; rdfs:domain :System ; rdfs:range :Component . :partOf owl:inverseOf :hasComponent . # Example :AppTier :hasComponent :AppServer1, :AppServer2 .
Pitfall: Don't confuse partOf with subClassOf — a wheel is part of a car, not a subclass of car.

Role Pattern

Use when an entity plays a context-dependent role. When to use: persons with job roles, servers in different environments, services acting as producers or consumers.

:Engineer rdf:type owl:Class . :OnCallEngineer rdfs:subClassOf :Engineer . # Role holder pattern: Person plays role :playsRole rdf:type owl:ObjectProperty ; rdfs:domain :Person ; rdfs:range :Role . :alice :playsRole :OnCallEngineer .
Pitfall: Avoid making roles permanent class assignments — a person can change roles without becoming a different entity.

Event Pattern

Use for time-bound occurrences with participants and affected resources. When to use: incidents, deployments, maintenance windows, outages.

:Incident rdf:type owl:Class . # Timeline properties :startedAt rdfs:domain :Incident ; rdfs:range xsd:dateTime . :resolvedAt rdfs:domain :Incident ; rdfs:range xsd:dateTime . # Participants & affected resources :hasResponder rdfs:domain :Incident ; rdfs:range :Person . :affects rdfs:domain :Incident ; rdfs:range :Service .

Mereology — Containment Hierarchies

Formal study of part-whole relationships with transitivity. When to use: datacenters containing racks containing servers, services containing microservices.

# Transitive containment — automatically chains :containedIn rdf:type owl:ObjectProperty, owl:TransitiveProperty . # If server is in rack, and rack is in DC, server is in DC :AppServer1 :containedIn :Rack42 . :Rack42 :containedIn :DatacenterWest . # Inferred: :AppServer1 :containedIn :DatacenterWest
Pitfall: Transitive properties can cause performance issues in large graphs — use carefully with reasoners.

Worked example: ITSM ontology

Start with a clear question list (competency questions): "Which servers host mission-critical applications?", "What is the impact radius of server S going down?", "Who is on-call for P1 incidents affecting service X?" These questions drive every modelling decision.

Define core classes: Server → hosts → Application → dependsOn → Database → affectedBy → Incident. Add data properties: hasIPAddress, hasStatus, hasPriority, startedAt. Add axioms: every Incident affects at least one Service; a P1 Incident must have an OnCallEngineer assignee.

Build in Protégé: create the class hierarchy, add object properties with domain/range, assert axioms, run HermiT reasoner to verify consistency. Export as Turtle or OWL/XML.

Load into GraphDB or Apache Jena Fuseki. Run SPARQL queries against live CMDB data. Connect to your AI pipeline for ontology-enhanced RAG, AIOps alert correlation, or LLM grounding.

Version control: store your .ttl or .owl file in Git. Tag releases. Validate changes with SPARQL-based test queries before deploying to production. Treat the ontology like code — review, test, deploy.
