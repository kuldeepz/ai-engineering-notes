Foundation

Why AI systems need ontologies

Without a structured model of your domain, AI systems hallucinate relationships, miss structural connections, and cannot plan across dependent entities.

🤖

LLMs without ontology

Hallucinate relationships, invent cardinalities, and have no reliable model of your domain's structure — especially for proprietary enterprise data.

Ungrounded
🔍

RAG without ontology

Vector similarity finds semantically close chunks but misses structural relationships — "what depends on this server?" requires graph traversal, not cosine similarity.

Incomplete
🕸️

Agents without ontology

No world model means agents cannot plan multi-step actions across dependent entities — they don't know what affects what, so they guess or hallucinate chains of action.

Blind planning

Key insight: An ontology is ground truth for your domain. It gives AI systems a machine-readable model of what exists, how things relate, and what rules govern them — enabling accurate reasoning instead of probabilistic guessing.

RAG Enhancement

Ontology-enhanced RAG

Four patterns for using ontology structure to improve retrieval quality beyond pure vector similarity.

🗺️  Graph-guided retrieval▼

Instead of only retrieving the most similar chunks, traverse ontology relationships to pull in structurally related content. If the query is about "CRMApp outage", also retrieve chunks about its database, the server it runs on, and dependent services.

When to use: Any domain where entities have dependencies — ITSM, codebases, supply chains, biomedical knowledge.

High precision
✂️  Entity-aware chunking▼

Instead of splitting documents by token count, split by ontological entities. A chunk about "Server X" stays together rather than being split mid-sentence, preserving the semantic unit the ontology defines.

When to use: When your documents are structured around domain entities — incident reports, product docs, service runbooks.

Better boundaries
🔎  Ontology-based query expansion▼

Before retrieving, expand the user query with related concepts from the ontology. Query "database slow" → ontology lookup → expand with related classes: storage layer, connection pool, query cache — then retrieve across all expanded terms.

When to use: Technical domains with rich synonym and relationship structures. Especially useful for non-expert users querying expert knowledge.

Better recall
⚡  SPARQL + vector hybrid▼

Use SPARQL for structured fact retrieval (exact relationships, counts, filters) and vector search for fuzzy semantic retrieval. Merge results with Reciprocal Rank Fusion (RRF). SPARQL answers "which services depend on Server X?" — vector answers "what documentation is relevant to this error?"

# Hybrid: SPARQL structured + vector semantic def hybrid_retrieve(query, entity): # Structured: get dependencies from ontology sparql_results = kg.query(f""" SELECT ?service WHERE {{ ?service :dependsOn :{entity} . }}""") # Semantic: vector similarity search vector_results = vectordb.search(query, top_k=10) # Merge with RRF return reciprocal_rank_fusion(sparql_results, vector_results)

Knowledge Graphs

Knowledge graphs — ontology at scale

A knowledge graph is an ontology populated with millions of real-world instances. The ontology defines the schema; the KG adds the data.

🏢

Enterprise KG

Internal company data modelled as a graph — employees, projects, systems, customers, incidents. Enables cross-silo search and relationship queries impossible with siloed databases.

Internal
🌐

Open KG

Publicly available knowledge graphs: Wikidata (100M+ entities), DBpedia (Wikipedia as RDF), Freebase (now Google KG). Used for entity linking, disambiguation, enrichment.

Public
🔬

Domain KG

Deep specialist knowledge: SNOMED CT (clinical terms), UniProt (proteins), ITIL ontology (ITSM), legal ontologies. Built with domain experts — high accuracy, narrow scope.

Flat CMDB vs ontology-backed CMDB: A flat CMDB stores relationships as database rows — you can query direct links but cannot traverse chains or infer new relationships. An ontology-backed CMDB supports reasoning: if Server A hosts App B which App C depends on, the system can automatically infer that a Server A outage impacts App C — without a hardcoded rule.

AIOps

Semantic alert correlation with ontology

Topology ontologies enable impact radius calculation, root cause isolation, and child alert suppression — reducing thousands of alerts to a handful of actionable incidents.

Rule-based correlation

  • → Hardcoded time windows
  • → Manually defined CI groups
  • → Breaks when topology changes
  • → Cannot infer transitive impact
  • → High maintenance overhead

Ontology-based correlation

  • ✓ Traverses live dependency graph
  • ✓ Infers transitive impact automatically
  • ✓ Updates as topology changes
  • ✓ Suppresses child alerts by inference
  • ✓ Explainable reasoning path
# SPARQL: find all services affected by a server outage PREFIX itsm: <http://example.org/itsm#> SELECT ?affectedService ?severity WHERE { # Direct: services hosted on the failed server { itsm:prod-server-01 itsm:hosts ?affectedService . } UNION # Transitive: services that depend on hosted services { ?midService itsm:dependsOn ?app . itsm:prod-server-01 itsm:hosts ?app . ?affectedService itsm:dependsOn ?midService . } ?affectedService itsm:businessImpact ?severity . } ORDER BY DESC(?severity)
LLM Grounding

Grounding LLMs with ontology

Three patterns for using ontology knowledge to make LLM outputs accurate and domain-correct.

Inject ontology into system prompt

Best for small ontologies (<200 classes). Serialise the ontology as natural language or Turtle and include in the system prompt. The LLM then uses it as ground truth when answering.

# Convert ontology triples to natural language for LLM def triples_to_nl(triples): lines = [] for s, p, o in triples: lines.append(f"- {s} {p.replace('_',' ')} {o}") return "\n".join(lines) system_prompt = f"""You are an ITSM assistant. Domain knowledge: {triples_to_nl(ontology.get_core_triples())} Only answer based on the above relationships."""
Simple · Fast

RAG over ontology triples

For large ontologies, embed each triple as a natural language sentence and store in a vector database. Retrieve relevant triples at query time and inject into context. Scales to millions of triples.

# Index ontology triples as embeddings for s, p, o in ontology.triples(): sentence = f"{s} {p} {o}" embedding = embed(sentence) vectordb.upsert(sentence, embedding, metadata={"s":s,"p":p,"o":o}) # At query time: retrieve relevant triples def get_ontology_context(query): relevant = vectordb.search(query, top_k=20) return "\n".join([r.text for r in relevant])
Scalable · Dynamic

Fine-tune on ontology-derived QA

Generate question-answer pairs from the ontology and fine-tune a smaller LLM on them. The model bakes domain relationships into its weights. Best for stable, slow-changing ontologies.

# Generate QA pairs from ontology triples qa_pairs = [] for s, p, o in ontology.triples(): q = f"What does {s} {p.replace('_',' ')}?" a = f"{s} {p.replace('_',' ')} {o}." qa_pairs.append({"question": q, "answer": a}) # Fine-tune with generated pairs fine_tune(base_model, qa_pairs, epochs=3)
Baked-in · Latency-free
Practical Guide

Building a domain ontology for AI

A pragmatic 5-step approach — start small, validate early, iterate with domain experts.

1
Identify entities — List the 10–20 most important things in your domain. For ITSM: Server, Application, Database, Incident, Change, Person, Team.
2
Define relationships — For each pair of entities, ask: how can A relate to B? Write these as verb phrases: hosts, dependsOn, assignedTo, triggeredBy.
3
Add constraints — Write competency questions: "Which servers host mission-critical apps?" If your ontology can't answer it, add the missing relationship or axiom.
4
Populate with instances — Load real data: CI records from CMDB, service catalogue, asset register. Use NER + relation extraction pipelines for unstructured data.
5
Connect to AI pipeline — Wire the knowledge graph into your RAG retriever, LLM context builder, or agent's tool set. Version control the ontology in git like code.

What this architecture gives you

Accuracy: LLM answers are grounded in your ontology's verified relationships — not hallucinated. When the LLM says "AppServer1 is critical infrastructure", it's because the ontology inferred it from real data, not because the model guessed.

Explainability: Every answer can be traced back through the SPARQL query and ontology path that produced it — essential for compliance, audit, and trust in enterprise AI systems.

Freshness: The knowledge graph is updated from live data sources. The LLM context reflects current state — not stale training data. This is especially critical for ITSM, security, and operational AI systems.

Scale: The ontology is defined once. New entities are added as instances. Reasoning rules apply automatically to all new data without retraining the LLM or updating prompts.
