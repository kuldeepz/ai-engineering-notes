Section 01

Two ways to give an LLM your knowledge

Both approaches solve the same problem — a base model doesn't know your data. They solve it in fundamentally different ways, with different tradeoffs.

🔍

Retrieval-Augmented Generation

Retrieve at query time
  • → Knowledge stored in a vector database
  • → Relevant docs fetched per query
  • → Injected into the LLM's context
  • → Model weights never change
Dynamic knowledge Cheap to update
🧠

Fine-tuning

Bake knowledge into weights
  • → Model trained on your data
  • → Knowledge baked into weights
  • → No retrieval step needed
  • → New model checkpoint produced
Style control Low latency
⚡

Hybrid Approach

Both together
  • → Fine-tuned model as base
  • → RAG for dynamic knowledge
  • → Best accuracy possible
  • → Highest cost and complexity
Best accuracy High complexity
RAG Analogy

Open book exam — you don't memorise everything, you look up the right page when you need it.

Fine-tuning Analogy

Closed book exam — you studied so thoroughly the knowledge is in your head, not on a page.

Section 03

5 questions to pick the right approach

Click each question to see the reasoning and which approach wins.

01 Does your knowledge change frequently? ▾

If your data is updated daily, weekly, or even monthly, RAG is almost always the right choice. With fine-tuning, you would need to retrain the model every time knowledge changes — this costs thousands of dollars and takes hours to days. With RAG, you simply re-index new documents into your vector database.

RAG wins — update in minutes, not days
02 Do you need to change the model's output style or format? ▾

Style, tone, and format are baked into the model's weights. If you need the model to always output JSON, respond in a specific brand voice, or follow a strict template, prompt engineering can only go so far. Fine-tuning internalises these patterns so you don't waste tokens on instructions every call.

Fine-tuning wins — style = weights, not prompts
03 Is privacy or latency critical? ▾

If you cannot send data to an external API (healthcare, legal, finance), a fine-tuned local model eliminates the external call entirely. For latency, a fine-tuned model on a local GPU removes the retrieval round-trip. That said, modern RAG pipelines with fast vector stores (sub-10ms) are often "fast enough" for most applications.

Fine-tuning wins for strict privacy / latency SLAs
04 How much labelled training data do you have? ▾

Fine-tuning requires labelled prompt-completion pairs. With fewer than 1,000 examples, fine-tuning often produces unstable or mediocre results — RAG with few-shot prompting will outperform it. With 10,000+ high-quality examples, fine-tuning starts to clearly win for task-specific accuracy.

Under 1K examples → RAG
Over 10K examples → Fine-tuning
05 What is your budget? ▾

RAG setup costs: embedding model + vector DB (~$50-200/month for moderate load). Fine-tuning a 7B model: ~$20-100 per run on cloud GPUs. Fine-tuning GPT-3.5 Turbo via API: ~$8/1M tokens. Full fine-tuning a 70B model: $500-2000 per run. A well-built RAG pipeline often delivers 80% of the value at 10% of the cost.

RAG wins on total cost for most use cases

Section 05

When RAG is the right choice

Four scenarios where RAG consistently outperforms fine-tuning.

🔄

Dynamic Knowledge

Docs, pricing, or policies that change regularly. New information available same-day without any retraining cycle.

RAG wins
🗂️

Multi-source Knowledge

Querying across Confluence, PDFs, databases, and APIs simultaneously. RAG handles heterogeneous sources naturally.

RAG wins
🔗

Explainability Required

Users or auditors need to see exactly which source produced which answer. RAG makes citations trivial.

RAG wins
📉

Limited Training Data

Fewer than ~1,000 labelled examples. Fine-tuning would overfit or hallucinate; RAG retrieves accurately.

RAG wins
Section 06

When fine-tuning is the right choice

Four scenarios where baking knowledge into weights beats retrieval.

🎨

Style / Tone Consistency

Legal writing, brand voice, medical notation — when every response must match a strict style that prompts alone cannot reliably enforce.

Fine-tuning wins
⚡

Latency-Critical Paths

Real-time applications (voice, gaming, trading) where the retrieval round-trip adds unacceptable latency to the critical path.

Fine-tuning wins
🔒

Strict Data Privacy

Healthcare, defense, or finance where data cannot leave a secure enclave. A local fine-tuned model has no external calls.

Fine-tuning wins
📋

Task-specific Output Format

Always outputting structured JSON, SQL, or code. Fine-tuning teaches the model the exact schema without wasting context on format instructions.

Fine-tuning wins

Section 08

End-to-end decision guide

How to walk through the decision in practice.

Step 1 — Start with RAG. The majority of production LLM applications should start with RAG. It is cheaper, faster to iterate, and does not require labelled training data. Build a solid retrieval pipeline first.

Step 2 — Measure where RAG fails. Run offline evaluation with a golden test set. Identify whether failures are retrieval failures (wrong docs fetched) or generation failures (right docs, wrong answer). Retrieval failures → improve chunking, embedding model, or re-ranker. Generation failures → consider fine-tuning.

Step 3 — Fine-tune only if justified. You need fine-tuning if: (a) style is inconsistent despite good prompts, (b) latency SLA cannot be met, (c) privacy prevents external API calls, or (d) you have 10k+ labelled examples and accuracy matters more than cost.

Step 4 — For fine-tuning, start with LoRA/PEFT. Full fine-tuning is almost never necessary for most tasks. LoRA trains ~0.5-1% of parameters, costs 10x less, and achieves ~90% of the accuracy gain. Use it for 7B-13B models on consumer or cloud GPUs.

Step 5 — Use hybrid only when you've validated both halves work. Don't build a hybrid system until you've proven RAG alone and fine-tuning alone separately. Combining them without a baseline makes it impossible to attribute improvements or failures.
