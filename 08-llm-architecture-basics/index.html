Section 02

Tokenization — how text becomes numbers

Everything the model sees is integers. Understanding tokenization explains why models struggle with character-level tasks and why some languages are more expensive.

Hover the tokens below to see their integer IDs (GPT-2 style tokenizer):

Phrase: "The quick brown fox jumps over the lazy dog"

The #464 quick #2068 brown #9060 fox #21831 jumps #18045 over #625 the #262 lazy #16931 dog #3290

9 words → 9 tokens (English maps cleanly). The same sentence in a lower-resource language might be 15-25 tokens.

BPE Tokenization

Byte Pair Encoding starts with individual characters and merges the most frequent pairs iteratively. Produces a vocabulary of ~50,000 subword units. Used by GPT models and most modern LLMs.

Why token count matters

You pay per token in API calls. Context limits are in tokens, not words. A 128K context window ≈ 96,000 words. Approximately 1 English word = 1.3 tokens on average.

1 word ≈ 1.3 tokens

Language efficiency gap

English is well-represented in training data — it tokenizes efficiently. Chinese, Arabic, and many other languages require more tokens per word, increasing cost and reducing effective context.

Tokenization quirks

LLMs struggle with counting characters or reversing strings because they operate on tokens, not individual characters. "strawberry" is ["str", "aw", "berry"] — the model can't easily count the 'r's.

Section 03

Attention mechanism — how the model connects words

Self-attention is the breakthrough that made modern LLMs possible. It allows every token to directly relate to every other token, regardless of distance.

Attention grid: darker cells = stronger attention between token pairs (example for "The cat sat on the mat")

Thecatsatonmat

Q, K, V — the three projections

Each token is projected into three vectors: Query (what am I looking for?), Key (what do I offer?), Value (what information do I carry?). Attention score = softmax(Q·Kᵀ / √d) · V

Multi-head attention

Instead of one attention pattern, the model runs H parallel "heads" with different projections. Each head learns a different type of relationship: syntax, semantics, coreference, positional patterns.

KV Cache — why inference is fast

During generation, the Key and Value matrices for previous tokens don't change. Caching them avoids recomputing the full attention for every new token. KV cache is why inference scales well for long contexts.

Critical for speed
# Simplified self-attention (scaled dot-product) import torch import torch.nn.functional as F def self_attention(Q, K, V): d_k = Q.shape[-1] # Q·Kᵀ gives us similarity scores between all token pairs scores = torch.matmul(Q, K.transpose(-2, -1)) / (d_k ** 0.5) weights = F.softmax(scores, dim=-1) # normalise to probabilities return torch.matmul(weights, V) # weighted sum of values
Section 04

Transformer architecture — layer by layer

A transformer consists of N identical layers stacked on top of each other. Click each component to understand what it does.

Input Token + Positional Embeddings ▾

Tokens are converted to dense vectors (embeddings) of dimension d_model (e.g. 4096 for LLaMA-3 8B). Positional encodings are added to tell the model where each token is in the sequence. Without position info, "dog bites man" and "man bites dog" would look identical.

Layer ×N Multi-head Self-attention ▾

Each token attends to every other token in the context. H parallel heads (e.g. 32 heads in LLaMA-3 8B) each learn different relationship patterns. Results are concatenated and projected back to d_model. This is the most computationally expensive part — O(n²) in sequence length.

Layer ×N Add & Layer Norm ▾

A residual connection adds the input to the attention output (prevents vanishing gradients in deep networks). Layer normalisation then stabilises training by normalising across the feature dimension. These are critical for training 100B+ parameter models stably.

Layer ×N Feed-forward Network (FFN) ▾

Two linear layers with a non-linearity (GELU or SiLU) between them. Projects to 4× d_model then back. This is where most factual knowledge is believed to be stored. In LLaMA-3 8B: d_model=4096, FFN intermediate=14336. Another residual + layer norm follows.

Output Language Model Head ▾

A final linear layer projects from d_model to vocab_size (~32,000 to 128,000). Softmax converts to probabilities over the entire vocabulary. The next token is sampled based on these probabilities and temperature. Then everything repeats for the next token.

Section 05

Key parameters — temperature, top-p, context window

The knobs you turn when using an LLM. Understanding them helps you predict and control model behaviour.

Temperature scales the logits before softmax. Higher temperature = flatter distribution = more random.

0 — Deterministic 1 — Balanced 2 — Chaotic

Current: 1.0

Temperature = 0

Always picks the highest-probability token. Fully deterministic. Use for tasks needing consistent, correct outputs: code generation, data extraction, classification.

Temperature = 0.7–1.0

Balanced creativity and coherence. Good for most chat applications, writing assistance, and summarisation where some variation is desirable.

Temperature > 1.0

Increasingly random. Useful for brainstorming and creative writing. Above 1.5 the output often becomes incoherent. Almost never useful for production systems.

Top-p (nucleus sampling) restricts token selection to the smallest set of tokens whose cumulative probability exceeds p. At p=0.9, only tokens in the "90% probability mass" are considered.

How it works

Sort tokens by probability. Keep adding tokens until cumulative sum exceeds p. Sample only from this "nucleus." This dynamically adjusts the number of candidates based on the distribution shape.

Typical values

p=0.9 to 0.95 is the most common production setting. Combining top-p with temperature (e.g. temp=0.7, top-p=0.9) gives fine-grained control. They can conflict — usually set one or the other.

The context window is the maximum number of tokens the model can process in a single forward pass — including both the prompt and the generated response.

Important: attention is O(n²) in sequence length. Doubling the context roughly 4× the attention computation. Long-context models use tricks like ring attention, sparse attention, and sliding window attention to manage this.

Top-k limits token sampling to the K most probable tokens at each step, regardless of their probabilities. A simpler alternative to top-p.

When to use top-k

k=50 is a common default. Works well when the probability distribution is fairly flat. Can produce unexpected results if k cuts off genuinely probable tokens in a peaked distribution.

Top-k vs top-p

Top-p is generally preferred because it adapts to the distribution shape. When the model is confident (one token has 80% prob), top-p naturally restricts more than top-k does.

Section 08

How to choose and use the right LLM for a task

A practical decision process for selecting and configuring an LLM for production use.

Step 1 — Define the task and output format precisely. Is this classification, generation, extraction, or reasoning? What does a "correct" output look like? This shapes every subsequent decision including model selection, temperature, and evaluation approach.

Step 2 — Start with the cheapest model that could plausibly work. Run your benchmark against GPT-4o mini, LLaMA-3 8B, or Mistral 7B first. Only upgrade to larger, more expensive models when a smaller model provably fails. Most tasks don't require the largest model.

Step 3 — Set temperature by task type. Extraction, classification, and code generation: temperature 0. Writing and summarisation: 0.5-0.8. Brainstorming: 0.8-1.0. Do not use temperature > 1.0 in production without a very specific reason.

Step 4 — Estimate token usage before writing a line of code. Count prompt tokens for typical requests. Multiply by expected daily request volume and price per million tokens. Add context window headroom for long conversations. Budget surprises at scale are painful.

Step 5 — Use the KV cache and batching. For repeated system prompts, place static content at the beginning of the prompt (cache-friendly). Batch multiple requests together when latency allows. These two optimisations often cut inference costs by 50-70%.

Step 6 — Monitor and iterate. Log every request and response. Track latency percentiles, error rates, and token counts. Measure quality with an offline golden set weekly. LLM APIs change and model quality drifts with new versions — continuous eval is non-negotiable.
