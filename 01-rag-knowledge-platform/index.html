Foundation

What is RAG and why does it matter?

RAG solves the biggest problem with LLMs in enterprise — they don't know your internal data. RAG gives them access without retraining.

🧠

The problem with pure LLMs

GPT-4 knows the world up to its training cutoff. It knows nothing about your internal KB articles, tickets, or SOPs. If you ask it about your company's IT policies, it will hallucinate.

Hallucination risk
🔍

What RAG does

Before answering, RAG retrieves the most relevant documents from your knowledge base. Then passes them to the LLM as context. The LLM answers using real, current, internal data.

Grounded answers
⚡

Why not fine-tuning?

Fine-tuning bakes knowledge into the model permanently. When your KB articles change, you retrain. RAG updates instantly — add a document, it's searchable in minutes. Much better for ITSM.

Dynamic knowledge
Architecture

The two pipelines — click each step

RAG has two distinct flows — ingestion (offline, one time) and query (real time, every request). Click any step to see the detail.

📥 Ingestion pipeline (offline)
STEP 01
Load documents
PDFs, Word, KB articles, wikis
Load all your source documents. In ITSM this includes KB articles, SOP documents, incident reports, policy docs. Use document loaders — LangChain has 80+ loaders for PDF, DOCX, HTML, Confluence, SharePoint. Extract clean text, preserve metadata (title, date, source URL, category).
↓
STEP 02
Chunk documents
Split into meaningful pieces
Documents are too large to embed whole. Split them into chunks of 256–512 tokens. Strategy matters — see the chunking section below. Always add overlap (10–15%) between chunks so context is not lost at boundaries. Store metadata with each chunk so you can filter later.
↓
STEP 03
Generate embeddings
text-embedding-ada / Azure OpenAI
Convert each chunk to a dense vector (array of numbers) that represents its meaning. We used text-embedding-ada-002 at Ivanti — 1536 dimensions. For large scale (50L+ docs) consider smaller dimension models (256–768) to reduce storage and search cost. Batch your embedding calls — Azure OpenAI allows 2048 chunks per batch.
↓
STEP 04
Store in vector DB
Index for similarity search
Store the embedding + original chunk text + metadata in a vector database. The DB builds a special index (HNSW or FLAT) that makes similarity search fast. For enterprise ITSM we used Azure AI Search — it combines vector search with traditional keyword search (hybrid search) which gives better results than vector-only.
🔎 Query pipeline (real time)
STEP 01
User asks a question
Natural language query
Employee types a question — "How do I reset a locked account?" or "What is the SLA for P1 incidents?" This is the entry point. You may want to preprocess the query — clean it, detect language, check for PII before sending to the LLM pipeline.
↓
STEP 02
Embed the query
Same model as ingestion — critical
Convert the user's question to an embedding using the exact same model used during ingestion. This is critical — if you use a different model, the vectors are in different spaces and similarity search fails. This call happens in real time and takes ~50ms with Azure OpenAI.
↓
STEP 03
Similarity search
Find top-K most relevant chunks
Search the vector DB for the top 5–10 chunks most similar to the query embedding. Similarity is measured by cosine distance — how close the vectors are in high-dimensional space. For enterprise ITSM, use hybrid search — vector similarity PLUS keyword matching. This handles cases where exact technical terms matter (e.g. "VPN-2024" won't be found by semantic search alone).
↓
STEP 04
Build prompt + call LLM
Context + question → answer
Combine: system instructions + retrieved chunks + user question into one prompt. Send to GPT-4o or Azure OpenAI. The LLM generates an answer grounded in the retrieved chunks. Critical rule — instruct the LLM to say "I don't know" if the answer is not in the retrieved chunks. Never let it hallucinate from its training data.
↓
STEP 05
Return answer + sources
Grounded, verifiable response
Return the answer with citations — which KB articles were used. This is critical for enterprise trust. Users can click through and verify. It also helps catch when the system retrieved wrong documents. At Ivanti we showed the top 3 source articles alongside every AI answer.
Chunking strategies

How to split documents — and why it matters

Chunking is the most underestimated part of RAG. Bad chunking = bad answers, even with a perfect LLM. Click each strategy to see it in action.

Overlap — the hidden key

Without overlap, answers that span two chunk boundaries get missed. With 10–15% overlap, every boundary is covered. This is a senior-level detail that matters significantly at scale.

Scale

What changes at 50 lakh+ documents

The architecture changes significantly at scale. Understanding these differences is critical for production deployments.

Scale Documents Embedding strategy Vector DB choice Key challenge
Small Up to 10,000 Sync, single batch Azure AI Search, Chroma Getting started fast
Medium 10K – 5 lakh Async queue, batch jobs Azure AI Search, Weaviate Incremental updates, stale docs
Large 5L – 50 lakh Distributed pipeline, Kafka Elasticsearch + vectors, Qdrant Search latency, index size, cost
Massive 50 lakh+ Streaming pipeline, sharded Elasticsearch sharded, Milvus Multi-tenancy, cost control, freshness
⚡

Batch embedding pipeline

At 50L+ docs, you cannot embed one by one. Use Kafka or Azure Service Bus to queue documents. Workers pick up batches of 2048 chunks. Run in parallel across multiple workers. Estimated time — 50 lakh documents ≈ 12–18 hours on 10 parallel workers.

🔄

Incremental updates

When a KB article is updated, don't reindex everything. Use document versioning — store doc_id and updated_at with each chunk. On update, delete old chunks for that doc_id and reindex only the changed document. This is critical for ITSM where KB articles change frequently.

💰

Cost control at scale

At 50L documents, embedding costs matter. text-embedding-ada-002 costs $0.0001 per 1000 tokens. 50L documents × 400 tokens avg = 200 billion tokens ≈ $20,000 for one full reindex. Use smaller models (text-embedding-3-small) or open source (sentence-transformers) to reduce cost.

Evaluation

How do you know RAG is working?

Evaluation is the most overlooked part of RAG. Without it, you have no idea if the system is working correctly in production.

F↑
Faithfulness
Is the answer grounded in the retrieved context? Does it hallucinate facts not in the source documents? Measure with RAGAS framework.
AR↑
Answer relevance
Does the answer actually address what the user asked? A faithful but off-topic answer is useless. Score with LLM-as-judge.
CR↑
Context recall
Did the retrieval step find all the relevant chunks? If the right KB article exists but wasn't retrieved, the answer will be wrong regardless of LLM quality.
P50↓
Latency (p50/p95)
End-to-end time from query to answer. For ITSM service desk, target under 3 seconds. Embedding + search + LLM call all add up. Monitor p95 not just average.
H↓
Hallucination rate
Percentage of answers that contain facts not in source documents. Target under 2% in production. Monitor with a golden test set of known Q&A pairs.
$/Q↓
Cost per query
LLM tokens + embedding tokens + search calls. At scale this matters. At 1 million queries/day even $0.001 per query = $30,000/month. Track and optimise.
End-to-end solution guide

Complete end-to-end solution

A complete walkthrough of the full RAG solution from ingestion to answer generation, in plain English.

Full solution walkthrough

"Before I design, can I ask a few things? How many documents approximately — hundreds or millions? What formats — PDFs, KB articles? Do documents update frequently? Any data privacy constraints like GDPR?"

"Okay, so I would build two pipelines. First the ingestion pipeline — we load all documents, split them into chunks of around 300 to 500 tokens using paragraph-based chunking with 10% overlap, convert each chunk to embeddings using Azure OpenAI's text-embedding model, and store everything in a vector database. For enterprise ITSM I'd use Azure AI Search because it supports hybrid search — vector plus keyword — which works better for technical terms."

"The query pipeline runs in real time. When an employee asks a question, we embed that query using the same model, do similarity search to find top 5 most relevant chunks, build a prompt with those chunks plus the question, and call GPT-4o. The LLM generates an answer grounded only in the retrieved documents. We always return citations so users can verify."

"At 50 lakh documents, the ingestion becomes a distributed pipeline — we'd use a message queue like Kafka, parallel embedding workers, and either Elasticsearch with vector support or Qdrant at that scale. Incremental updates are critical — when a KB article changes, we delete old chunks for that document and reindex only the changed one."

"For evaluation, I measure faithfulness, answer relevance, and hallucination rate using the RAGAS framework, plus latency at p95 and cost per query. We built something very similar at Ivanti for our Knowledge Base Article Generation feature and this architecture worked well in production."

# Simplified RAG pipeline — Python pseudocode

def ingest_document(doc_path):
  chunks = chunk_by_paragraph(doc_path, overlap=0.15)
  embeddings = azure_openai.embed(chunks, model="text-embedding-ada-002")
  vector_db.upsert(chunks, embeddings, metadata)

def answer_query(user_question):
  q_embedding = azure_openai.embed([user_question])
  top_chunks = vector_db.hybrid_search(q_embedding, k=5)
  prompt = build_prompt(user_question, top_chunks)
  answer = azure_openai.chat(prompt, model="gpt-4o")
  return answer, top_chunks # always return sources
