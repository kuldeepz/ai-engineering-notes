Foundation

What is ticket classification and why does it matter?

In enterprise ITSM, thousands of support tickets arrive daily. Routing them to the right team manually is slow, error-prone, and doesn't scale.

🎫

The routing problem

A user submits: "I can't log in to the VPN and my laptop is running very slow." Is this Access Management, Network, or Hardware? Getting this wrong means the ticket sits with the wrong team — multiplied by 10,000 tickets a day.

Routing errors cost time
🤖

What classification does

An ML model reads the ticket text and predicts the correct category, priority, or team in milliseconds. The ticket is routed automatically — no human triage for standard cases. Agents focus on complex, ambiguous tickets only.

Sub-second routing
📊

What you're predicting

Category (Network / Access / Hardware / Software), Priority (P1–P4), Assignment Group, and Sentiment. Each is a separate classification task — or a multi-label model predicting all simultaneously.

Multi-label prediction

How it works: Maintain a keyword list per category. If ticket contains "password" or "login" → Access. If "crash" or "BSOD" → Hardware. Rules are hand-crafted by domain experts.

When to use: Fewer than 10 categories and consistent, predictable ticket language. Good for a rapid prototype or as a high-confidence override layer.

Limitations: Breaks with synonyms ("can't get in" vs "access denied"), multi-topic tickets, and new terminology. Maintenance becomes a full-time job at scale. Real-world accuracy: 60–70%.

Still useful: As a fallback for deterministic edge cases ("password reset" → always Access team) and as a pre-filter before the ML model runs.
How it works: Convert ticket text to a TF-IDF vector (word frequency weighted by corpus rarity). Train LinearSVC or Naive Bayes on labelled tickets. Trains in seconds, inferences in under 5ms.

When to use: 5,000–100,000 labelled tickets, fewer than 50 categories. LinearSVC specifically is surprisingly strong — often matches deep learning on small ITSM datasets while being far simpler.

Limitations: No semantic understanding — "Network is NOT the problem" and "Network problem" look similar. Struggles with misspellings and language variations. Accuracy plateau: ~82–86%.

Why LinearSVC over Naive Bayes: LinearSVC handles multi-class better, is robust to feature scale, and produces reliable decision boundaries. Naive Bayes is faster to train but lower accuracy on imbalanced data.
How it works: Use a pre-trained sentence transformer (SBERT or OpenAI embeddings) to convert each ticket to a 768-dim semantic vector. Then train a LinearSVC on those vectors. This is the production sweet spot for most ITSM setups.

Why this combination wins: SBERT captures "can't log in" and "access denied" as nearly identical vectors. LinearSVC classifies fast. The full pipeline trains in minutes on a laptop. Accuracy: 88–94% on ITSM data with 1,000+ samples per class.

LSTM (older approach): LSTM reads word sequences and learns context. Outperforms TF-IDF but slower to train and requires more data than SBERT+LinearSVC. The SBERT approach replaces LSTM for most use cases.

When to choose SBERT over OpenAI embeddings: SBERT runs locally (no API cost, no latency, no data leaving your infrastructure). For 50L+ tickets this matters — OpenAI embedding cost at that scale becomes significant.
Zero-shot: Send ticket text + category list to GPT-4. No training data needed. Works immediately on new categories. Great for prototyping or handling rare ticket types.

Few-shot: Include 2–3 examples per category in the prompt. Significantly improves accuracy without any model training. Practical for up to ~20 categories.

Limitations: Cost ($0.002–0.01 per ticket), latency (1–3s vs <10ms for LinearSVC), privacy risk (ticket content leaves your network), output format variability.

The hybrid pattern that works in production: Use LinearSVC+SBERT for 80% of tickets (confidence > 0.75). Route low-confidence tickets to GPT-4. Best of both — low cost for bulk, high accuracy for edge cases.

Data challenge

The imbalanced data problem — and how to fix it

In real ITSM data, "Password Reset" might have 50,000 tickets while "Storage Failure" has 200. A naive model will predict the majority class for everything. Here is how to handle it.

Typical ITSM ticket distribution
⚖️

Class weights (apply this first)

Tell the model minority class errors are more costly. class_weight='balanced' in sklearn calculates weights inversely proportional to class frequency. Free, zero extra code, often sufficient on its own.

Always apply this
🔼

Oversample minority classes

For text: use back-translation (English → German → English) or LLM paraphrasing to generate synthetic variations of minority class tickets. Effective when you have fewer than 100 real examples for a class.

For <100 samples per class
🔽

Undersample the majority

Cap dominant classes at a max (e.g. 5× minority class count). Combined with oversampling in a hybrid approach. Use when the majority class is so large that training is slow and class_weight alone isn't enough.

For 100:1+ ratios
🛑

Minimum class threshold

If a class has fewer than 50 samples, don't train it as a standalone category. Merge it into a parent ("Other Hardware") or use LLM zero-shot for that class only. A model trained on 20 samples is unreliable in production.

Hard guard rail
Evaluation

How to measure classifier quality — per-class metrics

Accuracy is useless with imbalanced data. A model that always predicts "Password Reset" gets 40% accuracy — which sounds acceptable but it's completely wrong on every other class. Per-class F1 reveals the truth.

Precision
TP / (TP + FP)
Of all tickets labeled "Network" by the model, how many were actually Network? Low precision = false alarms. The Network team wastes time on irrelevant tickets.
Recall
TP / (TP + FN)
Of all actual Network tickets, how many did the model catch? Low recall = Network tickets misrouted to wrong teams. P1 incidents reach wrong engineers.
F1 Score
2 × P × R / (P + R)
Harmonic mean of precision and recall. The number to optimise for imbalanced datasets. Report per class, not just overall average — the average hides weak classes.
Macro F1
avg(F1 per class)
Unweighted average of F1 across all classes. Treats a rare class equal to a common class. The primary metric to track for ITSM classification. Target: >0.82 in production.
Per-class F1 — example production model

Notice how minority classes (Storage Failure) have much lower F1. These are the ones to investigate first — more data, better augmentation, or LLM fallback for that class.

Confidence threshold routing

Auto-route only tickets with model confidence > 0.75. Send low-confidence tickets to a human review queue. These become the best training examples — they're the ambiguous edge cases the model hasn't learned yet.

📉

Drift detection signals

Monitor weekly: if agent correction rate on auto-routed tickets rises above 15%, trigger a retrain. Also monitor new vocabulary appearing in tickets that wasn't in the training corpus — signals a new incident type or product launch.

🆕

New category protocol

New product or team? Collect 50+ labelled examples first. Add the class with class_weight compensation. Evaluate new class F1 before deploying. For <50 examples, use LLM zero-shot for that category while collecting data in the background.

Scale

What changes at 50 lakh tickets

The pipeline that works for 10,000 tickets needs architectural changes at 50 lakh. Throughput, cost, and retraining cadence all become critical.

ScaleVolumeRecommended approachLatency targetKey concern
Small < 50K TF-IDF + LinearSVC or SBERT + LinearSVC < 100ms Getting accuracy right, building labelled dataset
Medium 50K – 5L SBERT + LinearSVC, async queue, weekly retrains < 50ms Feedback loop, model freshness, drift monitoring
Large 5L – 50L Streaming pipeline (Kafka), batched embeddings, ONNX model serving < 20ms Throughput, multi-tenant isolation, retraining cost
Massive 50L+ Distilled BERT, cached embeddings, distributed serving, A/B testing < 10ms Cost per classification, GPU vs CPU tradeoff
⚡

ONNX for production speed

Export your trained LinearSVC + embedding model to ONNX format. ONNX runtime is 2–5× faster than sklearn at inference. Critical when processing 1,000+ tickets per second at peak load.

🧪

A/B test model versions

Never replace production model directly. Route 5% of traffic to new model, compare F1 on agent-corrected tickets after 48 hours. If new model wins, increase gradually. If F1 regresses, roll back immediately.

💾

Cache embeddings

At 50L tickets, 40–60% are variations of common requests ("password reset", "VPN issue"). Cache embeddings using a hash of normalized text. Reduces embedding compute cost by 40–60% and cuts latency for repeated patterns.

End-to-end solution guide

Complete solution walkthrough

The full production approach from data assessment to deployed classifier with feedback loop — in one coherent flow.

Full solution walkthrough

"First assess the existing data — labelled ticket count, number of categories, class distribution. This determines the approach. For most ITSM setups with 10,000+ labelled tickets, the sweet spot is SBERT embeddings with LinearSVC."

"Training pipeline: clean ticket text (remove ticket IDs, agent signatures, email threads), encode with all-mpnet-base-v2 SBERT model, train LinearSVC with class_weight='balanced' to handle imbalance. Evaluate with macro F1 and per-class F1 — never accuracy on imbalanced data."

"For deployment: wrap in a REST API. On each incoming ticket, embed it, classify, return category + confidence score. If confidence is below 0.70, route to human review queue — these become training examples for the next retrain."

"For new categories: collect 50+ labelled examples first, then retrain. Evaluate the new class F1 specifically before deploying. If you can't wait for data, use GPT-4 zero-shot for that category only while collecting examples in the background."

"Monitor weekly: agent correction rate, per-class F1 on production traffic, new vocabulary in tickets. Retrain when correction rate exceeds 15% or a new issue type emerges. The feedback loop is what keeps the system accurate as the business evolves."

# Production ticket classification — core pattern

class TicketClassifier:
  def __init__(self):
    self.encoder = SentenceTransformer('all-mpnet-base-v2')
    self.clf = LinearSVC(class_weight='balanced')
    self.threshold = 0.70

  def predict(self, text: str) -> dict:
    vec = self.encoder.encode([text])
    label = self.clf.predict(vec)[0]
    score = self.confidence(vec)
    if score < self.threshold:
      return {"label": "REVIEW_QUEUE", "confidence": score}
    return {"label": label, "confidence": score}

  def retrain(self, tickets: list, labels: list):
    vecs = self.encoder.encode(tickets)
    self.clf.fit(vecs, labels) # includes agent-corrected data
