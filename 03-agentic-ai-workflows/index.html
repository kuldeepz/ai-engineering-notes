01 — Foundation

What is an AI agent vs a plain LLM call?

An LLM responds. An agent acts. The difference is a loop, tools, and memory — turning a single prediction into a goal-directed system.

🧠

LLM Call

Single prompt → single response. No state, no tools, no follow-up. Stateless by design. Great for classification, summarization, and generation tasks that fit in one turn.

Stateless · One-shot
🤖

Agent

Loop + tools + memory. The LLM reasons about what action to take, executes a tool, observes the result, and loops until the goal is reached or a stopping condition fires.

Loop · Tools · Memory
🕸️

Multi-Agent

Multiple specialized agents coordinate toward a shared goal. One orchestrator delegates subtasks to worker agents, aggregates results, and handles failures across the network.

Coordination · Delegation
02 — The Agent Loop

The ReAct loop — think, act, observe, repeat

ReAct (Reasoning + Acting) is the dominant agentic pattern. The LLM alternates between thinking about what to do and actually doing it. Click each step to expand.

1
Thought — Reason about the goal and current state
+

The LLM receives the user's goal plus all previous observations and produces a scratchpad thought. It reasons: "I need to look up the ticket count. I should call the search_tickets tool with status=open."

This step is internal — the user never sees it. It gives the model space to plan before committing to an action. Longer reasoning here reduces hallucinated tool calls.

2
Action — Select and invoke a tool
+

The LLM outputs a structured action: which tool to call and with what arguments. The orchestrator parses this and dispatches the tool call. Tools can be API calls, code execution, database queries, web search, or any deterministic function with a clear schema.

The model cannot take multiple simultaneous actions in a single step — parallelism requires architectural support (fan-out patterns).

3
Observation — Receive and parse tool result
+

The tool result is injected back into the context window as an observation. The LLM now has new information to reason with. Raw tool outputs should be summarized or truncated before injection to avoid context overflow.

Error messages from tools are also observations — the agent can retry with corrected parameters or decide the task is infeasible.

4
Loop termination — When does the agent stop?
+

Three stopping conditions:

Goal reached — The LLM produces a final answer with no further tool calls.
Max steps exceeded — A hard cap (e.g., 15 iterations) prevents infinite loops. Return a partial answer.
Unrecoverable error — Tool consistently fails or returns permission denied. Agent surfaces the error to the user.

03 — Memory Types

Three kinds of memory in agentic systems — click each

Agents need memory to avoid repeating work and to build on prior context. Each memory type has different latency, cost, and persistence characteristics.

In-context Memory

Everything the agent has seen in the current session lives in the context window. This includes the original user goal, all Thought/Action/Observation turns, and any injected documents.

Stores
Current conversation, tool results, scratch reasoning
When to use
Single session tasks, <50 tool turns, fits in context budget
Limits
Token cap (4k–200k), lost when session ends, costly to fill

External Memory — Vector Database

Semantic facts are encoded as embeddings and stored in a vector store (Pinecone, Weaviate, pgvector). The agent retrieves relevant chunks via similarity search before reasoning.

Stores
Domain knowledge, past documents, product data, user preferences
When to use
Knowledge base QA, long-running agents, shared multi-session memory
Limits
Retrieval latency adds ~100–500ms per turn; recall ≠ perfect

Episodic Memory — Past Sessions

Summaries of past completed agent runs are stored and retrievable. When a new task starts, related past episodes are injected as context — "Last time you asked about X, the agent found Y."

Stores
Task summaries, user corrections, final answers, failed approaches
When to use
Personalized agents, recurring tasks, agents that learn from mistakes
Limits
Stale memory causes outdated advice; summaries lose fidelity
04 — Tool Use

How agents use tools

Tools are the only way an agent can affect the world beyond generating text. Every tool has a schema: name, description, and parameter types. The LLM selects tools by matching goal to schema description.

🔍

Search Tools

Web search, vector DB retrieval, knowledge graph lookup. Returns unstructured or semi-structured text. Best for open-domain question answering and research tasks.

⚙️

Code Execution

Sandboxed Python/JS REPL. Agent writes and runs code, captures stdout/stderr as observation. Enables data analysis, calculation, and programmatic transformations.

🌐

API Calls

External service integrations: ticketing systems, CRMs, monitoring APIs. Require auth handling, rate limit awareness, and structured error responses that the agent can reason about.

🗄️

Database Queries

SQL or NoSQL queries via a tool wrapper. The LLM generates queries from natural language. Requires schema context in the system prompt to avoid hallucinated table/column names.

05 — Orchestration Patterns

Multi-agent coordination patterns — click each

Once a task exceeds a single agent's capability, coordination is needed. Each pattern has different latency, error propagation, and cost characteristics.

Each agent's output becomes the next agent's input. Simple, predictable, easy to debug. Error propagation is a risk — a bad output from Agent A corrupts all downstream agents.

Use when: tasks have a natural ordered pipeline, early stages produce deterministic output, latency is not critical.

Multiple agents run simultaneously on independent subtasks. Latency is bounded by the slowest agent, not the sum. Cost is higher but wall-clock time improves significantly.

Use when: subtasks are independent, speed matters more than cost, tasks are easily decomposable (e.g., analyze 5 regions simultaneously).

Supervisor / Worker

A supervisor LLM dynamically decides which worker to invoke, reviews results, and can retry or reassign tasks. Workers are specialized agents with narrow tool sets and focused system prompts.

Use when: tasks are complex and non-linear, the supervisor needs to adapt the plan based on intermediate results, workers have domain-specific capabilities.

Multiple agents propose answers; others critique. A judge selects or synthesizes the best response. Reduces hallucination and catches errors that a single agent would miss.

Use when: accuracy is critical, the task is ambiguous or high-stakes (legal, medical, financial), self-consistency checks are needed. Expensive — 3–5× token cost.

06 — Production Failures

6 ways agentic systems fail in production

Agents introduce failure modes that don't exist in simple LLM calls. Every production deployment must anticipate and handle these.

Critical

Infinite Loops

Agent retries a failing tool indefinitely or oscillates between two states. Prevent: hard step limit (≤20), detect repeated identical actions, exponential backoff on tool errors.

Critical

Cost Explosion

Runaway token consumption from long loops or large context injection. Prevent: per-run token budget, cost alerting at 80% threshold, context compression after N turns.

High

Tool Hallucination

Agent calls a tool with invented parameter values or invents a tool name that doesn't exist. Prevent: strict JSON schema validation, reject calls to undefined tools, log all tool invocations.

High

Context Overflow

Accumulated tool outputs exceed the context window. Prevent: summarize observations >2k tokens, rolling window compression, external memory offload after N turns.

High

Race Conditions

Two agents write to the same resource simultaneously in a multi-agent system. Prevent: resource locking, optimistic concurrency with version checks, immutable shared state.

Medium

Non-deterministic Outputs

Same input produces different tool call sequences across runs due to LLM sampling. Prevent: temperature=0 for tool selection, seed the model where supported, log full traces.

08 — End-to-End Guide

Complete agent system walkthrough

Putting it all together — from receiving a user goal to returning a grounded, tool-backed answer with full observability.

Step 1 — Goal intake. Receive the user's natural language goal. Classify intent, check if it requires tools or can be answered directly. If tool use is needed, initialize the agent loop with a system prompt that includes tool schemas, memory context, and a step budget.

Step 2 — Memory retrieval. Query the vector store for relevant past episodes or knowledge chunks. Inject top-K results into the context window before the first Thought step. This grounds the agent in domain knowledge before it begins reasoning.

Step 3 — ReAct execution. Run the Thought → Action → Observation loop. After each observation, compress results that exceed 1000 tokens. Increment the step counter. On each action, validate the tool schema before dispatching. Log everything with a trace ID.

Step 4 — Output and storage. When the agent produces a final answer (no further tool calls), validate it passes a confidence threshold. Write the episode summary to episodic memory. Return the answer to the user with citations from tool results.

# Python pseudocode — production agent loop def run_agent(goal: str, max_steps: int = 15) -> str: memory = retrieve_memory(goal) # vector DB lookup messages = build_initial_context(goal, memory) step, cost = 0, 0.0 while step < max_steps: response = llm_call(messages, tools=TOOL_SCHEMAS) cost += estimate_cost(response) if cost > BUDGET_LIMIT: raise BudgetExceeded(cost) if response.finish_reason == "stop": save_episode(goal, response.content) return response.content # final answer tool_call = response.tool_calls[0] result = execute_tool(tool_call.name, tool_call.args) result = truncate_if_large(result, max_tokens=1000) messages.append({"role": "tool", "content": result}) step += 1 return "Max steps reached — partial answer: " + summarize(messages)
