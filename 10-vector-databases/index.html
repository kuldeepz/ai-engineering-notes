Section 01

What is a vector database and when do you need one

Traditional databases store and search by exact values. Vector databases store mathematical representations of meaning and search by semantic similarity. This is a fundamentally different paradigm.

📐

What a vector is

A dense array of floating-point numbers — e.g., [0.12, -0.87, 0.43, ...] with 768 to 3072 dimensions. Each dimension encodes some aspect of meaning learned by the embedding model. Semantically similar content produces geometrically close vectors.

High-dimensional float array
🔍

What similarity search does

Given a query vector, find the K vectors in the database that are geometrically closest — by cosine similarity or Euclidean distance. "Find nearest city on a map" — except the map has 1536 dimensions and contains millions of cities.

Nearest neighbour retrieval
⚡

When you need a vector DB

Use it when you need semantic search, RAG, recommendation, or deduplication across millions of records. Skip it when you have <50k docs (just use pgvector or in-memory), or when exact keyword matching is sufficient (Elasticsearch BM25).

RAG · Semantic search · Recommendations
Section 02

How vector search works

From storing embeddings to returning the nearest neighbours — click each step to see the technical detail.

1
Store vectors — ingest and index embeddings
⌄

What happens: You pass each document through an embedding model (e.g., text-embedding-3-large). The resulting float array is stored alongside metadata (doc ID, text, filters). The index builds an internal graph or cluster structure to allow fast retrieval later.

Key decisions: Choice of embedding model (dimension, language support), normalization (L2 normalize for cosine similarity), and whether to store the raw text or just the vector.

2
Query embedding — convert the question to a vector
⌄

What happens: The user's query is passed through the same embedding model used at ingest time. You must use the same model — vectors from different models are not comparable, even if they have the same dimensions.

Latency consideration: Embedding a short query typically takes 20–80ms via API. For ultra-low latency, run the embedding model locally or keep it warm in a sidecar container.

3
ANN search — approximate nearest neighbour lookup
⌄

What happens: An exact search would compare the query vector against every stored vector — O(n×d). For millions of vectors this is too slow. Approximate Nearest Neighbour (ANN) algorithms like HNSW trade a small accuracy loss for orders-of-magnitude speed improvement.

Accuracy vs speed: At ef=64, HNSW returns ~97% of the true nearest neighbours at 10× the speed of brute force. The "approximate" accuracy loss is almost never noticeable in real applications.

4
Return top-K with scores and metadata
⌄

What you get back: A ranked list of K results, each with a similarity score (0–1 for cosine, lower=closer for L2) and the metadata you stored at ingest time. You then pass these results as context to your LLM.

Score interpretation: Cosine similarity above 0.85 generally indicates high semantic relevance. Below 0.7 is often noise. Set a minimum score threshold rather than blindly passing all K results to the LLM.

Section 04

Search modes

Vector search and hybrid search solve different problems. Knowing when to use each is critical.

🧭

Pure vector search

Finds semantically similar content regardless of exact wording. "What is the return policy?" matches documents containing "refund procedure" even without keyword overlap.

Wins when: Paraphrased queries, multilingual search, conceptual similarity, ambiguous user language.

Loses when: Exact product codes (SKU-1234), proper nouns, technical acronyms — the embedding may not preserve these precisely.

🔀

Hybrid search (vector + BM25)

Runs vector search AND BM25 keyword search in parallel, then merges results using Reciprocal Rank Fusion (RRF). Gets the best of semantic understanding and exact matching.

RRF formula: score = Σ 1 / (k + rank_i) where k=60 is a smoothing constant and rank_i is each result's position in the individual ranked lists.

Default recommendation: Use hybrid for most RAG systems. The improvement in retrieval quality is worth the small latency overhead.

When hybrid wins over pure vector

Technical product documentation (part numbers, model IDs), customer support (ticket IDs, account numbers), code search (function names, variable names), medical records (drug names, ICD codes). In these domains BM25 catches exact matches that vectors smear across the embedding space.

Section 06

Filtering + vector search

Metadata filtering is essential for multi-tenant systems, date-range queries, and access control. The implementation strategy matters.

⚡

Pre-filtering

Apply metadata filter before the ANN search. Narrows the candidate set first, then runs ANN only on matching vectors. Faster but can miss relevant results when the filtered subset is very small — HNSW graph may not have enough connections within the subset.

🎯

Post-filtering

Run ANN search first, then filter results by metadata. Higher recall because ANN sees the full graph. More expensive — you must retrieve more candidates than K and filter down. Use when selectivity is high (>10% of index matches filter).

# Filtered vector search — Qdrant example from qdrant_client import QdrantClient from qdrant_client.models import Filter, FieldCondition, MatchValue, Range client = QdrantClient("localhost", port=6333) # Pre-filter: only search documents from tenant_id AND within date range results = client.search( collection_name="documents", query_vector=query_embedding, query_filter=Filter( must=[ FieldCondition(key="tenant_id", match=MatchValue(value="acme-corp")), FieldCondition(key="created_at", range=Range(gte=1704067200, lte=1735689600)) ] ), limit=10, with_payload=True, score_threshold=0.75 # ignore low-relevance results )

Section 08

Production patterns

Four architectural patterns every production vector DB deployment needs.

🏢

Namespace isolation (multi-tenant)

Each tenant gets their own namespace or collection. Prevents cross-tenant data leakage. Filter by tenant_id on every query — never rely on application logic alone to enforce isolation.

Hard isolation at query layer
🔄

Index refresh strategy

New documents: append immediately (HNSW supports incremental insert). Full re-index: needed when changing embedding model or chunking strategy. Use blue/green index swap to avoid downtime during re-index.

Zero-downtime re-indexing
💾

Backup and recovery

Snapshot the vector index and metadata store separately. Test restoration quarterly — vector index snapshots are large and restoration can be slow. Keep the source text/chunking pipeline so you can re-embed from scratch if needed.

RTO / RPO planning
📊

Monitoring (key metrics)

Query latency p95/p99 — spike indicates index fragmentation or resource pressure. Index size growth — plan for capacity. Recall@K — measure periodically with labelled test queries. Cache hit rate — semantic query cache reduces LLM calls.

Choosing your database: Start with Azure AI Search if you are on Azure and need hybrid search from day one. Use Qdrant if you want open-source with excellent performance and don't want to manage Kubernetes complexity. Use Chroma for local development and prototyping only. Use Milvus only when you genuinely need billion-scale collections and have ML infra engineers to operate it.

Choosing your index: Default to HNSW for all new production systems. Tune M=16 and ef_construction=200 as a starting point. Only consider IVF if your hardware budget is tight and you can accept slightly lower recall. FLAT is appropriate only for small collections or when you need guaranteed 100% recall.

The most common mistake: Skipping hybrid search. Pure vector search will miss exact product codes, error message strings, and proper nouns. Adding BM25 hybrid search typically improves RAG answer quality by 10–20% on real-world enterprise document sets with no change to the embedding model or chunking strategy.

Production readiness checklist: Tenant isolation enforced at query layer · Score threshold set (discard irrelevant results) · Index backup and restore tested · Recall@10 measured on labelled test set · Query latency p99 monitored · Embedding model version pinned (changing model requires full re-index).
