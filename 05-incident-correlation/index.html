01 — Foundation

What is incident correlation and why ITSM teams need it

Modern infrastructure generates thousands of events per second. Without correlation, every alert becomes a separate ticket. With it, 1000 alerts become 1 incident.

🌊

The Event Storm Problem

A single database outage triggers alerts from every service that depends on it — network, app, API gateway, load balancer, health checks. NOC teams receive 500–5000 alerts within minutes, all for the same root cause. MTTD explodes as operators drown in noise.

1 outage → 1000 alerts
🔗

What Correlation Does

Groups related alerts from different monitoring tools and infrastructure layers into a single parent incident. Surfaces the root cause alert and suppresses downstream symptom alerts. Operators see 1 incident instead of 1000 tickets.

1000 alerts → 1 incident
⚠️

Why Rule-Based Fails at Scale

Hand-written correlation rules require exhaustive knowledge of every possible alert combination. At 10k+ services, the rule space is combinatorially impossible to maintain. Rules break with every infrastructure change and cannot learn new patterns.

Unmaintainable at scale

Rule-Based Correlation
Predefined rules correlate alerts based on time windows, source topology, and alert type patterns. Example: "group all alerts from the same host within 5 minutes."

How it works: Alert A fires. A time window opens (e.g., 5 min). Any alert matching the rule's topology condition (same host, same VLAN, same service) is grouped under A.

Accuracy: High for known patterns, 0% for unknown patterns. Brittle — breaks on infrastructure changes.

Best for: Well-understood, stable environments with a limited alert taxonomy. Starting point for any correlation system.
03 — Architecture

The correlation pipeline — click each stage

Five stages transform a raw event stream into actionable, correlated incidents with identified root causes and suppressed noise.

Event Ingestion

Collect events from all monitoring sources into a single normalized stream. Events arrive from Nagios, Prometheus, Datadog, CloudWatch, SNMP traps, log-based alerting, and APM tools — all in different formats.

Normalization schema
Every event maps to: timestamp, source_host, source_tool, severity, alert_name, description, CMDB_CI_id, raw_payload. Missing fields are inferred from CMDB lookup.
Throughput considerations
Use a message queue (Kafka) as the ingestion buffer. Consumers normalize and enrich events with CMDB data. Target: sub-100ms end-to-end ingestion latency.

Deduplication

The same underlying alert often fires multiple times — from re-evaluations, duplicate monitoring tools, or alert flapping. Deduplication collapses these into a single event before correlation begins.

Dedup key
Hash of: source_host + alert_name + severity. Events with the same key within a 10-minute window are counted but not duplicated. Counter tracks frequency.
Flap detection
Alert that fires and clears more than 3 times in 15 minutes is flagged as "flapping." Flapping alerts are suppressed and create a single parent flap-detection ticket rather than N alerts.

Alert Grouping

Deduplicated events are grouped into candidate incidents using correlation algorithms. Grouping runs on overlapping sliding windows to catch slow-developing event cascades.

Grouping signals
Temporal proximity (same 5-min window), topology proximity (shared infrastructure dependency), semantic similarity (>0.8 cosine similarity of descriptions), historical co-occurrence (always correlated in past).
Group scoring
Each candidate group gets a confidence score (0–1). Score > 0.7 → auto-group and create incident. 0.4–0.7 → suggest to operator. < 0.4 → treat as separate incident.

Root Cause Identification

Within a correlated group, identify which alert represents the root cause vs downstream symptoms. The root cause alert is promoted as the incident title; symptoms are suppressed.

RCA signals
Earliest alert in the group by timestamp (first fire = likely cause), highest infrastructure layer (network > compute > app > service), CI dependency depth in CMDB, historical RCA labels from past incidents.
ML approach
Train a ranking model on labeled historical incidents (operators confirmed root cause). Features: temporal order, topology depth, alert severity, historical co-occurrence frequency.

Alert Suppression

Once a parent incident is identified, all child alerts are suppressed from creating new tickets. They're stored as evidence but do not page the NOC. Operators see 1 incident, not 50 tickets.

Suppression rules
New alerts matching active parent incident → suppress. Suppression window: parent incident is open + 5 minutes after resolution. Suppressed alerts appear in incident timeline.
Suppression risks
Over-suppression can hide new independent incidents. Guard with: max suppression count (100 per parent), independent-signal threshold, manual override by NOC operators.
04 — Temporal Correlation

Time-based pattern detection

Time is the primary correlation signal. Events close in time are likely related. Temporal analysis also detects bursts, cascades, and repeating patterns.

🪟

Sliding Window Approach

A time window (e.g., 5 minutes) slides over the event stream. All events within the window are candidates for grouping. Overlapping windows ensure no event cascade is split across window boundaries.

Window size is tunable: shorter (1 min) for fast-propagating failures, longer (15 min) for slow cascades across distributed systems.

💥

Burst Detection

An alert burst is a rapid spike in event volume within a short window. Bursts are the signature of cascading failures. Detect with: event count > 3× rolling average in any 60-second window.

Burst detection triggers aggressive correlation: all events in the burst window are forced into a single candidate group for RCA analysis.

🔁

Repeating Patterns

Some alerts fire on a repeating schedule — nightly batch jobs, weekend maintenance. These are classified as "known repeating" and suppressed automatically until their signature changes.

Pattern learned from 4+ weeks of history using time-of-week + alert-name fingerprint matching.

CMDB Integration

CI relationships from the CMDB are loaded into a graph database (Neo4j or in-memory adjacency list). Correlation engine traverses the graph to find all CIs affected downstream of a failing root CI.

Graph traversal

Impact Radius

Maximum hop count limits graph traversal (typically 3–4 hops). Beyond that, the relationship is too indirect to be a reliable correlation signal. Configurable per CI tier.

3–4 hop radius

Stale CMDB Risk

A CMDB that is not kept current causes missed correlations or false groupings. Validate CMDB accuracy monthly. Use network discovery tools to detect undocumented dependencies.

Maintenance critical

08 — End-to-End Solution

Complete correlation system walkthrough

From raw event stream to correlated incident with root cause — a production-grade AIOps correlation pipeline.

Step 1 — Event normalization. All monitoring tools publish events to a Kafka topic. A normalization consumer reads from the topic, enriches each event with CMDB CI data (host → CI_id, service tier, owner team), and writes the normalized event to the correlation engine state store. Any event missing a CMDB entry is flagged as "unknown CI" and treated independently.

Step 2 — Deduplication and flap detection. Events are keyed by (source_host, alert_name, severity). Events with the same key within 10 minutes increment a counter on the existing event rather than creating a new one. Events that fire and clear more than 3 times are classified as flapping and suppressed from correlation.

Step 3 — Correlation. A sliding window processor evaluates incoming events against three signals: temporal (same 5-minute window), topology (CI dependency graph within 3 hops), and semantic (cosine similarity > 0.8 on alert description embeddings). Groups with confidence > 0.7 are auto-promoted to incidents. Groups 0.4–0.7 are surfaced to operators for confirmation.

Step 4 — RCA and suppression. Within each group, the root cause candidate is selected using a ranking model: earliest timestamp + highest topology depth + historical RCA label weight. The root cause alert becomes the incident title. All other alerts in the group are suppressed — stored as evidence but not paging. The incident is routed to the owning team based on the root cause CI's assignment group in the CMDB.

# Incident correlation pipeline — Python pseudocode class CorrelationEngine: def __init__(self, cmdb: CMDBGraph, window_sec: int = 300): self.cmdb = cmdb self.window = window_sec self.open_groups: dict[str, IncidentGroup] = {} def process_event(self, event: NormalizedEvent): event = self.deduplicate(event) if event is None: return # duplicate — skip candidates = self.find_candidate_groups(event) best_group, score = self.score_candidates(event, candidates) if score >= 0.7: best_group.add_child(event) suppress_alert(event, parent=best_group.root) elif score >= 0.4: suggest_to_operator(event, candidate_group=best_group) else: new_group = IncidentGroup(root=event) create_incident(new_group) self.open_groups[event.ci_id] = new_group def score_candidates(self, event, candidates): best, best_score = None, 0.0 for group in candidates: t_score = temporal_score(event, group, self.window) topo_score = self.cmdb.dependency_score(event.ci_id, group.root.ci_id) sem_score = semantic_similarity(event.description, group.root.description) score = 0.4*t_score + 0.35*topo_score + 0.25*sem_score if score > best_score: best, best_score = group, score return best, best_score
