01 — Foundation

What is model drift and why does production ML degrade?

A model trained on last year's data is already becoming wrong. The world changes — inputs shift, relationships change, labels evolve. Drift is silent and cumulative.

📊

Data Drift

The statistical distribution of input features changes over time. The model was trained on one distribution but now sees a different one. Predictions may technically run, but quality quietly erodes.

Example: ticket language shifts after a product launch introduces new terminology the model never saw.

P(X) changes
🔄

Concept Drift

The relationship between inputs and the target label changes. The model's learned mapping no longer reflects reality — even if inputs look the same, the correct output is now different.

Example: "network down" tickets used to mean P1 but after a network upgrade, most are now P3 routine maintenance.

P(Y|X) changes
📉

Prediction Drift

The distribution of the model's output scores or labels changes, even if you can't observe true labels. A useful leading indicator — visible immediately without waiting for ground truth.

Example: P1 prediction rate drops from 15% to 3% after a holiday period with different incident patterns.

P(Ŷ) changes

Data Drift
Feature distributions shift over time. Word frequencies change, numeric ranges widen, new categorical values appear. The model inputs no longer resemble training data, so confidence scores become unreliable.

ITSM example: After a major cloud migration, ticket descriptions reference new cloud-native error codes that appeared 0 times in training data. The NLP model has no embeddings for these terms and defaults to the majority class.

Detection: PSI on numerical features, chi-square on categoricals, embedding distance for text. Threshold: PSI > 0.2 triggers alert.
03 — Detection Methods

How to detect drift statistically — click each

No single test catches all drift types. A production pipeline should run multiple complementary tests on each monitoring window.

PSI — Population Stability Index

Measures how much a numerical feature's distribution has shifted between a reference (training) window and a current production window. Originally from credit risk monitoring.

How it works
Bin both distributions into 10–20 equal-width or equal-frequency buckets. For each bucket: PSI_i = (Actual% − Expected%) × ln(Actual%/Expected%). Sum across all buckets.
Thresholds
PSI < 0.1 → No action
0.1 ≤ PSI < 0.2 → Investigate
PSI ≥ 0.2 → Significant drift, retrain

Works best for numerical features with clear distributions.
def compute_psi(expected, actual, buckets=10): bins = np.percentile(expected, np.linspace(0, 100, buckets+1)) exp_freq = np.histogram(expected, bins=bins)[0] / len(expected) act_freq = np.histogram(actual, bins=bins)[0] / len(actual) exp_freq = np.clip(exp_freq, 1e-6, None) act_freq = np.clip(act_freq, 1e-6, None) return np.sum((act_freq - exp_freq) * np.log(act_freq / exp_freq))

Kolmogorov-Smirnov (KS) Test

A non-parametric test that measures the maximum difference between two empirical cumulative distribution functions (CDFs). Sensitive to both shift and shape changes.

How it works
D = max|F1(x) − F2(x)| over all x. Returns a statistic D and a p-value. Large D and small p-value means the distributions are significantly different.
Thresholds
p-value < 0.05 → Statistically significant drift
D > 0.1 → Practical significance

Best for continuous numerical features. Requires sufficient sample size (≥ 1000 observations).
from scipy.stats import ks_2samp stat, p_value = ks_2samp(reference_data, production_data) if p_value < 0.05 and stat > 0.1: trigger_drift_alert("KS", stat, p_value)

Chi-Square Test

Tests whether observed categorical frequencies differ significantly from expected (training) frequencies. The go-to test for categorical features like ticket categories, priority levels, or department names.

How it works
χ² = Σ (Observed − Expected)² / Expected. Degrees of freedom = number of categories − 1. Large χ² means the production distribution differs from training.
Thresholds
p-value < 0.05 → Alert
p-value < 0.001 → Severe drift

Requires each category to have ≥ 5 expected observations. Merge rare categories before testing.
from scipy.stats import chisquare expected_props = reference_df['category'].value_counts(normalize=True) observed_counts = production_df['category'].value_counts() expected_counts = expected_props * len(production_df) chi2, p = chisquare(f_obs=observed_counts, f_exp=expected_counts)

Feature Importance Shift

Re-measure which features the model relies on most at production time vs training time. A significant ranking change indicates the model is making decisions based on different signals than intended.

How it works
Use SHAP values on a sample of production predictions. Compare the mean absolute SHAP value ranking to training-time rankings. Large rank correlation drop → feature importance drift.
Thresholds
Rank correlation (Spearman) < 0.8 → Investigate
< 0.6 → Significant shift

Computationally expensive — run weekly rather than real-time.

Prediction Distribution Monitoring

Monitor the distribution of output scores and predicted labels over time. This requires no ground truth labels and gives the fastest signal — visible within hours of deployment.

What to track
Mean confidence score per window, class frequency ratios, proportion of low-confidence predictions (<0.6), bimodal distribution detection in confidence scores.
Thresholds
Mean confidence drops > 0.1 → Alert
Class ratio shifts > 20% → Alert

Use rolling 7-day vs 1-day window comparison for seasonal awareness.
04 — Monitoring Pipeline

The production monitoring architecture

Every prediction is an opportunity to measure drift. A production pipeline logs, tests, alerts, and responds — continuously and automatically.

Feature Logger

Log every prediction with its input features, output score, predicted label, timestamp, and model version. Store in a time-series database. Log compression: store summary statistics per 1000 predictions, raw logs for debugging.

Every prediction

Statistical Tests

Run PSI, KS, chi-square on hourly or daily windows comparing against the training baseline. Schedule on rolling windows to avoid day-of-week seasonality false positives.

Hourly / Daily

Alert Thresholds

Two-tier alerts: yellow (0.1 ≤ PSI < 0.2, p-value 0.05–0.01) triggers investigation; red (PSI ≥ 0.2, p-value < 0.01) triggers automated response. Page on-call for red alerts.

Tiered severity

Response Automation

On red alert: auto-create investigation ticket, freeze model version deployment, notify model owner. On confirmed drift: trigger retraining pipeline or initiate rollback to last stable version.

Automated triage

🔁

Triggered Retraining

A CI/CD pipeline monitors drift metrics. When a threshold is crossed and new labeled data is available, retraining kicks off automatically. New model goes through shadow testing before promotion.

Event-driven
📅

Scheduled Retraining

Weekly or monthly retraining regardless of detected drift. Simpler ops, predictable cost. Suitable for slowly evolving domains. Combine with triggered approach for critical models.

Time-based
↩️

Rollback

Immediate revert to a previous model version when a data quality issue or pipeline failure causes sudden performance collapse. Must have versioned model artifacts stored and tested for each deployment.

Emergency
06 — Metrics to Track

Production observability metrics

These are the metrics that surface drift early and confirm model health. Track all of them — each catches different failure modes.

📈
Prediction Confidence Distribution
Mean and percentile distribution of model output scores per window. Sudden drop in mean confidence is an early drift signal.
Alert: mean drops > 0.1 vs 7-day baseline
📊
Feature Drift Score
PSI or KS statistic per feature, aggregated into a single composite drift score for dashboards. Tracks which features are drifting most.
Alert: composite PSI > 0.15
✅
Label Agreement Rate
When human-labeled ground truth becomes available (e.g., ticket resolution), compare to model prediction. Tracks actual accuracy against reality.
Alert: agreement drops > 5% vs 30-day baseline
⏱️
Inference Latency p95
95th percentile latency per model version. Latency spikes can indicate infrastructure drift — model running on different hardware or with increased payload size.
Alert: p95 > 2× baseline
❌
Prediction Error Rate
Rate of predictions that fail (timeout, exception, or invalid output). Increasing error rate signals infrastructure or input schema drift.
Alert: error rate > 1% in any 5-min window
🔀
Class Frequency Ratio
Ratio of predicted class frequencies vs training class frequencies. Catches label imbalance drift and concept drift simultaneously.
Alert: any class ratio shifts > 20%

08 — End-to-End Guide

Complete drift detection pipeline

From prediction logging through statistical testing to automated response — a complete production-grade drift detection system.

Step 1 — Instrument predictions. Every model inference logs: timestamp, input features (or a hash for PII safety), output score, predicted label, model version, and latency. Write to a time-series store partitioned by date. Enable sampling for very high-volume models (10% sample is sufficient for statistical tests at scale).

Step 2 — Compute baseline statistics. At training time, compute and store the reference distribution for every feature: mean, std, percentile bins, and categorical frequencies. Store PSI bin edges alongside the model artifact so tests are reproducible even if training data is deleted.

Step 3 — Run tests on schedule. Scheduled job (hourly or daily) pulls the current window, computes PSI and KS for numerical features, chi-square for categoricals, and prediction distribution shift. Emit all scores as metrics. Alert on threshold breach.

Step 4 — Respond. Yellow alert → auto-create investigation ticket, tag model owner. Red alert → halt new deployments, notify on-call, run automated data quality checks. If confirmed drift → trigger retraining pipeline with latest N days of labeled data, shadow test new model, promote when performance exceeds incumbent by ≥ 2%.
