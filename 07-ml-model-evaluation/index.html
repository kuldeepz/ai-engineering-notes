Section 01

Why evaluation is the hardest part of production ML

A model that passes offline evaluation can still fail in production. Evaluation is not a one-time gate — it is a continuous process at every stage of deployment.

📊

Offline Evaluation

Evaluated against a fixed dataset before deployment. Controlled, reproducible, but may not reflect real user behaviour. The baseline gate every model must pass.

Before deploy
📡

Online Evaluation

Measured with real users in production. A/B tests, shadow mode, canary deployments. The only way to measure true business impact and distribution shift.

After deploy
👤

Human Evaluation

Annotators or end-users judge model outputs directly. The gold standard for subjective tasks — but expensive, slow, and hard to scale. Used to calibrate automated metrics.

Ground truth
⚠️

The fundamental trap

Passing offline evaluation does not guarantee production quality. Real users ask different questions, provide edge-case inputs, and use the system in ways you never tested. Offline eval is necessary but never sufficient.

Section 03

RAG evaluation with RAGAS

RAGAS (Retrieval Augmented Generation Assessment) provides 6 automated metrics for evaluating RAG pipelines without requiring ground-truth labels for every response. Click a metric to expand it.

FaithfulnessTarget: >0.85
Are all claims in the answer supported by the retrieved context?

Measures whether the generated answer contains only information that can be directly inferred from the provided context. Hallucinated facts score 0 on this metric.

score = supported_claims / total_claims

If low: Improve chunking to include more complete context, or reduce temperature.

Answer RelevanceTarget: >0.80
How directly does the answer address the original question?

Uses an LLM to generate hypothetical questions from the answer, then measures cosine similarity between those and the original question. A tangential answer scores low.

score = mean_cosine_sim(generated_questions, original_question)

If low: Refine the generation prompt to stay focused on the question.

Context RecallTarget: >0.75
Did retrieval fetch all the documents needed to answer correctly?

Requires ground-truth answers. Checks what fraction of the sentences in the ground-truth answer can be attributed to the retrieved context. Measures retrieval completeness.

score = GT_sentences_in_context / total_GT_sentences

If low: Increase K (top-K docs), improve chunking, or use a re-ranker.

Context PrecisionTarget: >0.70
How much of the retrieved context is actually relevant?

Measures the signal-to-noise ratio of your retrieval. If you fetch 5 docs but only 1 is relevant, precision is 0.2. High noise increases LLM confusion and token cost.

score = relevant_chunks_retrieved / total_chunks_retrieved

If low: Use a re-ranker, reduce K, or improve your embedding model.

Answer CorrectnessTarget: >0.75
Is the answer factually and semantically correct vs ground truth?

Combines factual overlap (F1 on extracted claims) with semantic similarity to a reference answer. Requires labelled ground-truth answers. The most direct quality measure.

score = 0.75 * factual_F1 + 0.25 * semantic_sim

If low: Check all other metrics first — correctness is a downstream symptom.

Answer SimilarityTarget: >0.80
How semantically similar is the answer to the ground-truth answer?

Pure embedding-based cosine similarity between the generated answer and the reference. Captures paraphrase and semantic equivalence, not just exact token overlap like BLEU.

score = cosine_sim(embed(answer), embed(reference))

If low: The model may be answering correctly but in a very different style — check human eval.

# RAGAS evaluation setup from ragas import evaluate from ragas.metrics import (faithfulness, answer_relevancy, context_recall, context_precision) from datasets import Dataset data = { "question": ["What is RAG?"], "answer": ["RAG combines retrieval with generation..."], "contexts": [["RAG stands for Retrieval-Augmented Generation..."]], "ground_truth": ["Retrieval-Augmented Generation is a technique..."] } result = evaluate(Dataset.from_dict(data), metrics=[faithfulness, answer_relevancy, context_recall, context_precision]) print(result) # {'faithfulness': 0.92, 'answer_relevancy': 0.87, ...}
Section 04

LLM-as-Judge — using an LLM to evaluate LLM outputs

When there is no ground truth and human annotation is too expensive, an LLM can serve as a scalable judge. It requires careful calibration.

Question + Answer
→
Judge LLM (e.g. GPT-4)
→
Score 1–5
+
Reasoning

When to use

No ground-truth labels exist. Subjective quality (helpfulness, clarity, safety) matters. You have budget for a strong judge model (GPT-4o or Claude Opus) at evaluation time.

Prompt design for judges

Always include a rubric with anchor examples for each score. Specify exactly what to measure. Avoid vague instructions like "evaluate quality" — define quality explicitly.

Calibrate against humans

Run 100-200 examples through both the LLM judge and human annotators. Report Spearman correlation and Cohen's kappa. A good judge model achieves r > 0.8 correlation with experts.

Biases to watch for

Position bias (prefers first answer), verbosity bias (prefers longer answers), self-enhancement bias (prefers its own provider's outputs). Mitigate with randomised ordering and multi-judge averaging.

# LLM-as-Judge prompt template JUDGE_PROMPT = """You are evaluating an AI assistant response. Question: {question} Response: {response} Rate the response on HELPFULNESS (1-5): 1 = Does not address the question 3 = Partially addresses with minor errors 5 = Fully addresses, accurate, well-explained Output JSON only: {{"score": <1-5>, "reasoning": "<1-2 sentences>"}} """ def judge_response(question, response, judge_llm): prompt = JUDGE_PROMPT.format(question=question, response=response) result = judge_llm.generate(prompt) return json.loads(result)

👻

Shadow Mode

New model runs on every request but its response is discarded. Zero user impact. Collects latency, error rates, and outputs for offline comparison before any live traffic.

0% live traffic
🐦

Canary Deployment

Send 1-5% of real traffic to the new model. Monitor error rates, latency, and implicit signals (click-through, session length) before full rollout.

1–5% live traffic
⚖️

Full A/B Test

50/50 split for statistical power. Run until you have 95% confidence (typically 1,000-10,000 samples per arm depending on effect size). Use a two-proportion z-test.

50% live traffic
🎰

Multi-armed Bandit

Automatically routes more traffic to the better-performing variant. Reduces regret by exploiting good options faster, but makes it harder to interpret statistical results cleanly.

Dynamic allocation
Section 06

Human evaluation — when machines can't judge

Human evaluation is expensive but remains the ground truth for subjective quality. Use it strategically.

When to use human eval

Automated metrics don't capture what users care about. New task type with no established benchmark. Calibrating LLM-as-Judge. Regulatory or safety requirements demand it.

Annotation guidelines

Write a detailed rubric with 5+ anchor examples before annotation starts. Bad guidelines produce noisy labels that are worse than no labels. Pilot with 50 examples before full run.

Inter-annotator agreement

Measure Cohen's kappa or Fleiss' kappa across annotators. Below 0.6 = unreliable labels. Fix guidelines, not annotators. Always use at least 2 annotators per example.

Cost vs quality tradeoffs

Expert annotators: $50-200/hr, slow but high quality. Crowdsource: $0.05-0.20/task, fast but noisy. For production: use experts to calibrate, crowd to scale, LLM judge to run daily.

Section 07

Evaluation anti-patterns to avoid

Common mistakes that create a false sense of confidence in model quality.

Anti-pattern

Only measuring accuracy

Accuracy hides class imbalance failures. A model that always predicts the majority class can have 95% accuracy on an imbalanced dataset. Always use F1, AUC-ROC, or confusion matrix.

Anti-pattern

No production monitoring

Models degrade silently as data distributions shift. You need continuous monitoring of input distributions, output distributions, and downstream business metrics — not just launch-time eval.

Anti-pattern

Overfitting to benchmarks

Optimising exclusively for benchmark scores produces models that score well but fail in production. Benchmarks are proxies. Always maintain a holdout that the team cannot see during development.

Anti-pattern

Ignoring latency as a metric

A model with p99 latency of 10 seconds fails users even if accuracy is perfect. Track latency percentiles (p50, p95, p99) alongside quality metrics from day one.

Section 08

End-to-end evaluation pipeline

How all three evaluation types connect in a production ML system.

Phase 1 — Offline (pre-deploy). Build a golden test set of 100-500 curated examples. Run automated metrics (task-appropriate: F1, RAGAS, custom LLM-judge). Only move to Phase 2 if all metrics exceed thresholds.

Phase 2 — Shadow mode. Run new model in parallel for 24-72 hours on real traffic. Compare output distributions, latency, and error rates with no user impact. Identify surprises before any live exposure.

Phase 3 — Canary (1-5%). Expose a small fraction of real users. Monitor implicit feedback signals: task completion rate, re-query rate, session length. Run for 48-72 hours minimum.

Phase 4 — Full A/B test. 50/50 split until statistical significance (95% confidence). Primary metric is the business metric you care about, not the ML metric. Kill switch if error rate spikes.

Phase 5 — Production monitoring. Never stop evaluating. Track input distribution drift (KL divergence), output quality drift, and business metrics weekly. Trigger re-evaluation if any signal degrades by >5%.

# Minimal eval pipeline skeleton class EvalPipeline: def __init__(self, model, golden_set, metrics): self.model = model self.golden_set = golden_set self.metrics = metrics def run_offline(self) -> dict: preds = [self.model.generate(q) for q in self.golden_set["questions"]] return {m.__name__: m.score(preds, self.golden_set["answers"]) for m in self.metrics} def check_thresholds(self, results, thresholds) -> bool: return all(results[k] >= v for k, v in thresholds.items()) def should_deploy(self) -> bool: results = self.run_offline() thresholds = {"faithfulness": 0.85, "answer_relevancy": 0.80} return self.check_thresholds(results, thresholds)
