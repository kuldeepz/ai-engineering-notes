Foundation

Why AI projects are notoriously hard to estimate

Software estimation is hard. AI estimation is harder — by an order of magnitude. The inputs are unknown, the outputs are probabilistic, and the integration adds a third layer of uncertainty most estimates ignore completely.

🗃️

Data unknowns

Data quality, availability, labelling effort, and legal clearance are all unknown until you look. Each is a potential blocker that invalidates estimates made before data assessment.

Upstream unknown
🎯

Experiment uncertainty

You cannot know whether a model will reach target accuracy before running experiments. The model might hit 95% in week 2 or require months of iteration to reach 80%.

Probabilistic outcome
⚙️

Integration complexity

A working ML model is not a working product. Production engineering — APIs, latency optimisation, error handling, monitoring — often takes as long as the model itself.

Hidden work

Key principle: Give ranges, not point estimates. "4–8 weeks" is honest and defensible. "6 weeks" is a guess presented as a commitment. Stakeholders may push back on ranges — explain that a point estimate carries hidden uncertainty that will surface as a miss, not as a risk.

The 5 Phases

Every AI project has these phases — click each

These five phases appear in every AI project. Each has different uncertainty, different blockers, and different skills required. Collapsing or skipping phases is the most common source of blown timelines.

🔍

Data Assessment

1–3 weeks
+

What happens: Discover what data exists, profile data quality, identify gaps, assess labelling requirements, check legal / GDPR clearance.

Typical duration: 1–3 weeks depending on data access complexity.

Common blockers: Data behind access control gates, undocumented schemas, missing data for edge cases, legal review queues.

What adds time: Multiple data owners, no data dictionary, PII that requires anonymisation before use.

🧪

Experimentation

2–8 weeks
+

What happens: Baseline modelling, feature engineering, model selection, hyperparameter tuning, evaluation against defined success metrics.

Typical duration: 2–8 weeks — the most variable phase in any AI project.

Common blockers: Poor baseline data, class imbalance, target metric is not well defined, no labelled evaluation set.

What adds time: Target accuracy is too high for the available data, novel problem domain with no prior art to draw from.

🏗️

Engineering

3–6 weeks
+

What happens: Productionise the model — build API, integrate with product, optimise latency, add error handling, write tests, document behaviour.

Typical duration: 3–6 weeks for most ML features; longer for real-time or embedded systems.

Common blockers: Latency requirements not met by model, prediction service architecture not agreed, no feature serving infrastructure.

What adds time: Latency-constrained real-time serving, complex upstream data pipelines, integration with legacy systems.

🚀

Deployment

1–3 weeks
+

What happens: Staged rollout (shadow mode → canary → full), A/B test setup, monitoring dashboard, alerting rules, runbook documentation.

Typical duration: 1–3 weeks for shadow/canary + full rollout with monitoring.

Common blockers: Security review queues, infrastructure change freeze, no A/B testing infrastructure, compliance sign-off.

What adds time: Regulated industries requiring sign-off, high-risk model decisions (lending, healthcare), complex rollback requirements.

📡

Monitoring and Iteration

Ongoing
+

What happens: Drift detection, performance tracking, retraining triggers, model refresh cycles, feedback loop integration.

Typical duration: Ongoing — budget 20–30% of initial build cost per year for maintenance and retraining.

Common blockers: No monitoring budget allocated, production data distribution shifts, no retraining pipeline built during initial delivery.

What adds time: Data distribution shifts faster than expected, regulatory changes requiring re-evaluation, growing edge cases in production.

The Data Tax

Why data always takes longer than estimated

Data work is systematically underestimated in every AI project. It is the single biggest source of schedule overruns. Apply the multiplier before you commit.

🔎

Data discovery

Finding where data actually lives, who owns it, what schema it follows, and whether it contains what you think it contains. Often takes 2x longer than assumed.

🧹

Data quality

Missing values, label noise, distribution skew, duplicate records, stale data. Quality problems are never obvious until you run your first model and get confusing results.

🏷️

Labelling

Human annotation is slow, expensive, and disagreement between annotators is higher than expected. Budget 1 week per 1,000 labels for complex classification tasks.

⚖️

Legal / privacy clearance

GDPR, data residency, consent requirements, and third-party data agreements all require legal review. Legal queues can add 2–4 weeks to any project start date.

✕3

Rule of thumb: Whatever your data preparation estimate is — multiply it by 3. Not because engineers are wrong, but because data problems compound. The first problem reveals the second, which reveals the third. Budget accordingly before the project starts.

🔴

Before data assessment

Estimate cone is at its widest. A 5x range is normal and honest. Never commit to a delivery date before you have confirmed data availability and quality.

🟡

After data assessment

Cone narrows to roughly 2x. You now know what you are working with. This is the first point at which a sprint-level plan is meaningful.

🟢

After prototype

Cone narrows to 1.3x. You know the model architecture works, the data pipeline is built, and the main remaining work is engineering and deployment.

Risk Factors

What blows AI project timelines — click to see mitigation

Each of these risks has appeared in real AI projects. Click any risk card to flip it and see the mitigation strategy.

⚠️

Data quality worse than expected

The dataset looks fine until you train a model — then you find 30% of labels are wrong or the feature distributions are totally different from production.

click to see mitigation →
✅

Mitigation

Run a mandatory data assessment sprint before estimation. Profile all features. Check label agreement rate. Build a sample evaluation set and validate it with a domain expert before modelling starts.

⚠️

Model accuracy target unreachable

The business requirement is "95% accuracy" but the data supports 78% maximum. Neither side knew this at estimation time, so the project runs forever trying to hit an impossible target.

click to see mitigation →
✅

Mitigation

Set a human performance baseline before defining the target. Agree on minimum viable accuracy (what is the cost of error?) rather than aspirational accuracy. Create an accuracy milestone gate at week 3 of experimentation.

⚠️

Infrastructure not ready

The model is ready to deploy but the serving infrastructure, GPU allocation, or CI/CD pipeline for ML doesn't exist. Engineering work blocks on infra for 3–6 weeks.

click to see mitigation →
✅

Mitigation

Start infrastructure work in parallel with experimentation, not after. Define the serving architecture during the data assessment phase. If infra doesn't exist, add infra setup as an explicit project phase with its own timeline.

⚠️

Stakeholder requirement changes

Midway through an ML project, the business requirement changes — new target metric, new population to cover, new output format. The experiment phase restarts.

click to see mitigation →
✅

Mitigation

Lock the success metric definition before any experimentation starts. Document it as a formal acceptance criterion. Changes after experimentation begins require a re-estimate and re-scoping session — make this process explicit upfront.

⚠️

Security / compliance review delays

The model is ready. The security review queue is 6 weeks. Or the compliance team needs 3 weeks to review the data processing pipeline you are using.

click to see mitigation →
✅

Mitigation

Engage security and compliance during data assessment, not at deployment. Submit for preliminary review early. Build review lead times into the timeline explicitly — never assume they are instantaneous.

⚠️

Third-party API reliability

LLM API latency spikes unpredictably. Rate limits hit in production. The vendor changes pricing or deprecates a model version. Production stability suffers.

click to see mitigation →
✅

Mitigation

Design for fallback from the start — caching, retry logic, and model version pinning. Evaluate total API cost at expected production volume before launch. For critical systems, consider running an open-source alternative in parallel.

Stakeholder Communication

How to present AI estimates

Honest range-based estimates are more useful than false precision. Here is how to communicate them in a way that builds rather than erodes stakeholder trust.

↔️

Use ranges, not points

Present "6–10 weeks" as a confidence range, not a hedge. Explain which variables move you toward the shorter or longer end. This shows analytical rigour, not uncertainty.

🔬

Separate research from engineering

State clearly: "Experimentation phase is 2–5 weeks and outcome is uncertain. Engineering phase is 3–4 weeks and is certain once the model is confirmed." Do not blend them into one number.

🎯

Set accuracy milestones, not just ship dates

Define gate criteria: "At week 4, the prototype must reach at least 80% F1 to proceed to engineering. Otherwise, we re-scope." This manages the most common AI estimation failure mode.

🔄

Re-estimate at each phase gate

Build in explicit re-estimation checkpoints after data assessment and after prototype. Share the updated estimate with stakeholders — this is not a sign of planning failure, it is how AI projects work.

Example language: "We estimate 6–10 weeks total, once data quality is confirmed clean in the assessment sprint. If data requires significant cleaning, we are looking at 10–14 weeks. We will re-estimate at the end of data assessment (week 2) and again after the modelling prototype (week 6)."

Common Mistakes

Estimation mistakes that commit teams to impossible timelines

These four mistakes are the most common causes of AI projects running significantly over time and budget.

❌

No data assessment phase

Estimating the project without examining the data first means every assumption about data quality and availability is unvalidated. The most common source of 2x timeline overruns.

Most expensive mistake
❌

Estimating ML and engineering together

Combining uncertain research work with deterministic engineering work into a single estimate obscures where the uncertainty lives and makes re-estimation conversations harder.

Visibility risk
❌

No buffer for re-experimentation

The first model rarely works well enough. Budget at least 1.5x the initial experimentation estimate to cover iteration after the baseline — model improvements are rarely linear.

Iteration risk
❌

Single-point estimates under pressure

Stakeholders push for a single number. Engineers cave and give one. That number becomes a commitment. The range gets lost and the uncertainty goes underground until it resurfaces as a miss.

Process failure
Estimation Template

End-to-end estimation for a full AI project

A structured template for estimating any AI project from data assessment through to live monitoring. Use this as a starting point and adjust multipliers for your specific context.

Step 1 — Classify the project type: Identify whether this is classification, RAG, agentic, fine-tuning, or platform work. Each has a different phase distribution. Refer to the timeline tabs above for baseline ranges.

Step 2 — Run the data assessment first: Before committing to any timeline, spend 1–2 weeks assessing data availability, quality, and legal status. Only after this sprint can you give a meaningful estimate to stakeholders.

Step 3 — Estimate each phase separately: Produce a range for each phase: data prep, experimentation, engineering, deployment, and monitoring setup. Never present a single total without showing the phase breakdown.

Step 4 — Apply the data multiplier: Take your data preparation estimate and multiply it by 1.5–3 depending on data maturity. If data is greenfield (no existing pipelines, no labels), apply the 3x multiplier.

Step 5 — Document assumptions and risks: Every estimate is only valid under stated assumptions. Write them down — "this assumes 50k labelled examples are available, that serving latency requirement is under 200ms, and that no new data source integration is needed."

Final output format: "We estimate [X–Y weeks] assuming [named assumptions]. The biggest uncertainty is [named risk]. We will re-estimate after [named milestone, e.g. data assessment sprint / modelling prototype]. Key risks that could extend to [Z weeks]: [list risks with mitigation]."
